# 09 — Internationalisation (i18n)

The game is bilingual: Serbian (Latin) is the authored language, English is a translated overlay. This chapter is the reference for both layers — the in-browser language pack used by the engine, and the offline Gemini-based translator that produces the English content.

The conceptual placement of i18n in the architecture is in [04 §5](04-architecture.md#5-language-pack-pattern). This chapter is the implementer's manual.

## 1. Two layers of translation

The game has two distinct categories of text, each translated differently:

| Layer | Lives in | Shape | Used by |
|---|---|---|---|
| **UI strings** | `ui.js` | `UI = { sr: {key:val}, en: {key:val} }` | All static HTML via `data-i18n` attributes; engine-rendered text via `t("key")` |
| **Content** | `scenarios.js`, `personas.js` | `HCP_SCENARIOS = { sr: [...], en: [...] }` etc. | Engine dispatcher functions `getScenarios()`, `getProfiles()`, `getPersonas()` |

The shapes differ because UI is a dictionary (random access by key) while content is an ordered list (positional access by index). Both layers have the same fallback rule: **if the active language is missing or empty, fall back to Serbian.**

## 2. The lookup helpers

### `t(key, params)` — UI lookup

```js
function t(key, params) {
  const dict = UI[S?.lang] || UI.sr;
  let s = dict[key] ?? UI.sr[key] ?? key;
  if (params) for (const k in params) s = s.replace(`{${k}}`, params[k]);
  return s;
}
```

The three-step fallback is `UI[lang]` → `UI.sr` → the key itself (so missing keys are visible in dev rather than silent). Placeholder substitution is `{name}` style:

```js
t("step.counter", { n: 3 })  // → "Korak 3/4 · "
```

`tn(baseKey, n, params)` is a plural variant for Serbian's three forms (1 / 2–4 / 5+) — looks up `{baseKey}.1`, `{baseKey}.few`, or `{baseKey}.many` based on `n`. English falls through to two forms (singular / plural).

### `applyI18n(root)` — static HTML refresh

Walks the DOM under `root` (default `document`) and rewrites every element with a `data-i18n` attribute:

```html
<h2 data-i18n="role.title">U <em>čijim</em> cipelama danas?</h2>
```

After language change, `applyI18n()` replaces `innerHTML` with `t("role.title")` for the new language. The original Serbian text is kept in the HTML source as a no-JS fallback.

### `getScenarios()` / `getProfiles()` / `getPersonas()` — content dispatch

```js
function getScenarios() {
  const en = HCP_SCENARIOS?.en, sr = HCP_SCENARIOS?.sr;
  if (S.lang === "en" && en && en.length) return en;
  return sr || [];
}
```

The fallback is *array-level*, not item-level — if `HCP_SCENARIOS.en` exists and has at least one entry, all eight scenarios come from there, even if individual entries are placeholder `null`s. This matters during translation: a half-translated array could expose `null` items. In practice the translator's incremental write ensures the array is either complete-so-far or empty.

## 3. Where language is set

In priority order (`engine.js` `DOMContentLoaded` handler):

1. **`?lang=sr` or `?lang=en` URL parameter** — set by `qr.html` when generating a workshop QR. Wins over everything else for this session.
2. **`S.lang` from `localStorage`** — the user's last manual choice via the header toggle.
3. **Default `"sr"`** — if no URL param and no saved state.

The user can change language at any time via the header pills (`<button class="lang-pill" onclick="setLang('en')">EN</button>`). `setLang()` updates `S.lang`, persists, calls `applyI18n()` for static text, and calls `rerenderCurrentScreen()` to refresh dynamic content. The current screen ID is inferred from which `<section>` is not `.hidden`.

## 4. The Gemini-based translator

`scripts/translate.js` is a Node 18+ script with **zero npm dependencies**. It reads the Serbian content, calls the Gemini API with a careful prompt, validates the output structure matches the input, and patches the relevant `en:` block into the source file.

### 4.1 Setup

Once per machine:

```bash
cp .env.example .env
# Edit .env, fill in GEMINI_API_KEY=…
```

`.env` is gitignored. The script ships its own minimal `.env` parser (no `dotenv` dependency).

The default model is set in `.env`:

```
GEMINI_API_KEY=AIza…
GEMINI_MODEL=gemini-2.5-flash
```

You can override per-run with `--model <name>` (see §4.4).

### 4.2 Commands

```bash
node scripts/translate.js ui                # all UI strings
node scripts/translate.js profiles          # 4 HCP profiles
node scripts/translate.js scenarios         # all 8 HCP scenarios (incremental)
node scripts/translate.js scenario <idx>    # one HCP scenario (0–7)
node scripts/translate.js personas          # all 8 personas (incremental)
node scripts/translate.js persona <idx>     # one persona (0–7)
```

Flags:

- `--dry-run` — prints translated JSON to stdout, does not patch any file.
- `--model <name>` — override the model from `.env`.

### 4.3 What happens per command

1. **Load.** The Serbian source is read with Node's `vm` module — the script `require`s `ui.js` / `scenarios.js` / `personas.js` into a sandbox and extracts the named global (`UI`, `HCP_PROFILES`, `HCP_SCENARIOS`, `PERSONAS`).
2. **Call Gemini.** A single REST POST to `generativelanguage.googleapis.com/v1beta/models/<model>:generateContent` with `responseMimeType: "application/json"`. The prompt is the style guide (§4.5) plus the input JSON.
3. **Strip code fences.** Even with `responseMimeType: json`, some models wrap output in ```` ```json … ``` ```` — strip these before parsing.
4. **Validate.** `validateSameKeys()` recursively checks that the translated tree has identical keys (and array lengths) to the Serbian source. Throws if not.
5. **Patch.** A file-specific function rewrites the `en:` block in place (§4.6).
6. **Continue or stop.** Batch commands (`scenarios`, `personas`) write after *each* item, so a mid-batch failure leaves earlier items committed.

### 4.4 Model strategy and free-tier quotas

The Gemini free tier rotates limits frequently; the project has been calibrated for three models:

| Model | Status (2026-05) | Use when |
|---|---|---|
| `gemini-2.5-pro` | Hard limit `0` on free tier — requires billing. | Paid tier only. Best quality. |
| `gemini-2.5-flash` | 20 requests / day on free tier. | Default. Good quality, sufficient for the project's ~30 translatable items. |
| `gemini-2.5-flash-lite` | Separate quota from `flash`. | Fallback when `flash`'s daily limit is exhausted mid-batch. |

The default in `.env` is `gemini-2.5-flash`. If a batch run hits the daily limit (visible as `429` with a `RetryInfo` suggesting a 24-hour wait), switch:

```bash
node scripts/translate.js personas --model gemini-2.5-flash-lite
```

The translator's "skip already translated" logic (§4.7) means the resumed run only translates what's still missing.

### 4.5 The translation prompt

The script ships a fixed style guide (~70 lines, `STYLE_GUIDE` constant in `translate.js`) covering:

- **Structure rules.** Output only JSON. No prose, no markdown fences. Translate values; keep keys.
- **Preserved fields.** Persona names (Marija, Jelena, Sanja, Petar, Goran, Dragan, Tijana, Maja, Lana, Iva, Stefan, Lazar, Mila, Olja), doctor names and initials, `id` slugs, quality flags (`good`/`neutral`/`bad`), step type flags (`root`/`affirm`/`refute`/`facts`), ending keys (`best`/`rushed`/`delayed`/`mid`/`pending`/`closed`), unlock tags (`shared`/`family`/`deep`/`weak`/`high`/`low`), URLs, numbers.
- **HTML preservation.** `<em>`, `<strong>`, `<br>`, `<a onclick="…">` tags must remain identical in shape; only visible text content is translated.
- **Tone.** Natural, professional, empathic — not "translationese". Second-person where the Serbian uses second-person. Emotional weight (grief, fear) preserved.
- **Terminology glossary.** A fixed mapping for ~20 domain terms — `saslušanost` → `feeling heard`, `Eliciraj` → `Elicit`, `Afirmišite` → `Affirm`, `Opovrgnite` → `Refute`, `Činjenice` → `Facts`, `prihvatanje vakcinacije` → `vaccination acceptance`, etc.
- **Cultural-reference rules.** Locations and Big Pharma stay verbatim. Serbian healthcare-system context translated factually.
- **Plural conversion.** Serbian's three-form (1/2-4/5+) maps to English two-form (singular/plural).
- **Quote style.** Convert Serbian's `«…»` and `„…"` to standard English `"…"`.

Tuning the glossary is the most common reason to edit `translate.js`. Adding a new term is just one line — but re-running existing commands will rewrite previously translated text to match.

### 4.6 The three patchers

Each of the three target files has a different in-place edit strategy, because each has a different surrounding structure.

#### `replaceEnObject(filePath, enContent)` — for `ui.js`

`ui.js` has one `en:` block under `UI = { sr: {…}, en: {…} }`. The patcher finds `},\nen: {` using:

```js
src.match(/\},\s*(?:\/\/[^\n]*\n\s*)*en:\s*\{/)
```

…then bracket-counts `{` / `}` until depth returns to zero, and replaces the matched range with `en: <stringified new object>`.

**The cautionary tale.** Earlier versions of this regex were just `en:\s*\{`. The very first time the translator ran, it found a match — but in the JSDoc comment at the top of `ui.js` that documents the data shape (`UI = { sr: {…}, en: {…} }`). The English content was patched into a code comment instead of the actual `UI.en` object, and the file fell silently into a syntax error the next time it loaded. The fix was to require the `en:` to follow `},\n` (the closing brace of the `sr:` block plus optional intervening comments). The regex above is the result.

The lesson generalises: anything that pattern-matches into a JavaScript source file must be careful that strings inside comments are not matched. Bracket-counting alone is not enough.

#### `replaceEnArray(filePath, enContent, varName)` — for `personas.js`

`personas.js` has one `,en:[` block at the end of `PERSONAS`. The patcher finds the *first* `,en:[`, bracket-counts to the matching `]`, and rewrites the range. Used for `personas` and `persona <idx>` commands.

#### `patchScenariosEn(enContent)` — for `scenarios.js`

`scenarios.js` is the awkward case: it has **two** `,en:[` blocks (one for `HCP_PROFILES.en`, one for `HCP_SCENARIOS.en`) in the same file. `patchScenariosEn()` finds the *second* occurrence specifically. The first is handled separately by `replaceEnArray()` when the `profiles` command runs.

Future maintenance: if a third `,en:[` array is ever added to `scenarios.js`, this approach breaks. The pragmatic fix would be to switch from positional matching to a named regex like `HCP_SCENARIOS\s*=\s*\{[^]*?en:\[`.

### 4.7 Resilience features

**Incremental write.** `scenarios` and `personas` (the batch commands) save the patched file after *every successful item*. If the run dies on item 5 of 8 (rate limit, network blip, ctrl-C), items 0–4 are already committed.

**Skip already translated.** Before each item, the batch command checks whether `HCP_SCENARIOS.en[i]` / `PERSONAS.en[i]` is already a non-null object. If so, it logs *"already translated, skipping"* and moves on. To force re-translation of a single item, use the per-item command (`scenario <idx>`, `persona <idx>`) — those bypass the skip check.

**Exponential backoff retry.** Transient errors (`429`, `500`, `503`, `504`) trigger up to 5 attempts with delays of 4s, 8s, 16s, 32s, 60s. If the API response includes a `RetryInfo.retryDelay`, the script honors that instead (capped at 120s).

**Structure validation.** `validateSameKeys()` recursively walks the original and the translation in parallel. It checks:

- Arrays have the same length.
- Object keys are identical (set equality).
- Recursive sub-trees follow the same rule.
- *Values* are deliberately NOT checked — they should differ (that's the point of translation).

Any structural drift throws *before* the patch is written, so the file is never left in an inconsistent state.

## 5. The recommended workflow

When adding new Serbian content (a new scenario, a new persona):

1. Write and validate the Serbian version first. Play it in the browser. Don't worry about English at all.
2. Once the Serbian is what you want, run the per-item translate command:
   ```bash
   node scripts/translate.js scenario 8   # for a new 9th scenario
   ```
3. Switch the language toggle to EN and play through the new content in English. Look for awkward phrasing, mistranslated technical terms (the glossary should catch most), or unexpected line wrapping.
4. If a single string reads badly, hand-edit `HCP_SCENARIOS.en[8].steps[2].o[0].fb` directly. The translator will not overwrite it unless you re-run the per-item command.
5. Commit both files together. The English pack is part of the same change as the Serbian source.

When editing existing content (a tweak to one line):

1. Edit the Serbian. Test in browser.
2. Either re-run the per-item translate command (clean but uses an API call), or hand-edit the English equivalent (faster, no API call needed).
3. Commit.

## 6. What this does *not* support yet

- **Languages other than `sr` and `en`.** The shape supports any number of languages but the translator script is hard-coded for `en` as output. Chapter [08 §5](08-extending.md#5-add-a-third-language) walks through what extension would require.
- **Reverse translation** (EN → SR). The Serbian is the authored source. The script will not run in the other direction.
- **Per-string translation memory.** The translator re-translates everything in a batch from scratch. A future version could diff against the previous English and only re-call for changed strings — useful when the API has stricter quotas.
